# Animals and Water Physics Plan

## Goal

Add animals as believable ambient wilderness systems without turning Stick into a fast action or hunting game. The first animal prototype should be a fish, because fish force the game to model water as a gameplay volume rather than only a rendered surface.

This plan supports the task list item in [Tasks](tasks.md): add animals.

## Current State

Water currently exists as deterministic world-space features generated by `WorldFeatureGenerator`:

- lakes and ponds
- rivers
- carved lake/river terrain
- rendered water surfaces/ribbons

However, water is mostly visual and terrain-shaping data right now. There is no general gameplay water volume and no object physics for being in water.

Current limitations:

- Lake water is rendered with simple planes.
- River water has a smoother ribbon mesh, but is still prototype-grade.
- The player is vertically snapped to terrain height every frame.
- There is no buoyancy, drag, sinking, swimming, wading, current, or water occupancy state.
- Animals do not exist yet.

## Core Direction

Separate water rendering from water gameplay. Animals and player physics should not ask whether they are inside a Babylon mesh. They should query a gameplay water layer that describes the water column at a world position.

Recommended dependency flow:

```text
WorldFeatureGenerator
  ↓
Water volume/query layer
  ↓
Water-aware player/object physics
  ↓
AnimalSystem
  ↓
Fish prototype
```

## Water Gameplay Query

Add a lightweight water query service before or alongside the first fish prototype.

Implemented source location:

```text
src/world/water/WaterVolumeSampler.ts
```

Core API:

```ts
interface WaterColumnSample {
  readonly hasWater: boolean
  readonly featureId: string | null
  readonly type: "lake" | "river" | null
  readonly surfaceY: number
  readonly bedY: number
  readonly depthMeters: number
  readonly distanceToShoreMeters: number
  readonly flowDirectionX: number
  readonly flowDirectionZ: number
  readonly currentMetersPerSecond: number
}
```

Implementation inputs:

- `WorldFeatureGenerator.sample(worldX, worldZ)` for feature membership and water surface level
- terrain height sampling for bed height
- river centerline/profile data for approximate current direction

Water volume definition:

```text
water exists from bedY up to surfaceY
```

A point is submerged when:

```text
hasWater && worldY >= bedY && worldY <= surfaceY
```

Initial current behavior can be very simple:

- lakes: no current, or near-zero drift
- rivers: downstream direction from nearest river segment
- current speed based on river type/width/slope later

## Player Water Physics Direction

The player should eventually move from terrain-snapped vertical placement to a simple kinematic body with vertical velocity.

Short-term player states:

```text
grounded
airborne
wading
submerged
```

Suggested first behavior:

- On land, continue to follow/support terrain.
- In shallow water, wade and reduce movement speed.
- In deeper water, apply gravity, drag, and weak buoyancy.
- A person should not be neutrally buoyant by default; they should slowly sink without swimming/treading input.
- Swimming can be added later as explicit movement behavior.

Avoid adding a full physics engine unless the custom kinematic model becomes too limiting.

## Animal System Direction

Start with a small explicit OOP animal system rather than a large generic framework.

Possible source layout:

```text
src/animals/
  AnimalSystem.ts
  AnimalTypes.ts
  AnimalSpawner.ts
  FishController.ts
  FishMeshFactory.ts
```

`AnimalSystem` should be a `GameSystem` and receive dependencies through constructor injection:

- `EngineContext`
- player position provider
- water query service
- terrain height provider if needed
- world seed / feature data

Responsibilities:

- choose nearby runtime animal spawn candidates
- instantiate active animals near the player
- update active animal behavior
- dispose distant animal meshes
- keep animal logic independent from chunk persistence and terrain caches
- treat animals as dynamic in-memory actors; when the player leaves and later returns, old disposed animals should not be reloaded from cache

## Fish MVP

The first fish should be ambient and believable, not a major interaction system. Fish should be uncommon: eventually the player may be able to find and catch them, so spotting one near shore should feel fortunate rather than routine.

### Runtime / Persistence Rule

Fish and other animals should be dynamic in-memory actors, not cached terrain content. They should not be serialized into chunk snapshots or rehydrated when a chunk reloads. Leaving an area may dispose the active fish; returning may create new runtime fish, but those are fresh actors rather than cached entities.

### Spawn Rules

Spawn fish only where:

- `WaterColumnSample.hasWater === true`
- water depth is at least about `0.75m`
- position is inside the water volume, including occasional near-shore positions
- position is within an active radius of the player
- a low runtime spawn chance succeeds

Recommended first pass:

- Lakes and rivers can provide spawn-eligible water areas.
- Runtime spawning should occur near the player and remain independent from terrain chunk persistence.
- Only instantiate/render fish near the player.

### Behavior

Use simple steering rather than complex AI.

Fish should:

- wander within valid water volume
- maintain a target depth below surface and above bed
- avoid shore/invalid water
- avoid the player at close range
- drift gently with river current
- steer back into the water volume if close to the surface, bed, or shore

Depth constraints:

```text
minY = bedY + 0.15m
maxY = surfaceY - 0.15m
targetY = bedY + depthMeters * 0.45
```

For rivers:

- add mild downstream drift
- optionally face into or across current
- avoid leaving the channel

### Visual Prototype

Use a low-poly procedural placeholder first:

- small ellipsoid body
- triangular tail fin
- simple material
- yaw aligned to velocity
- subtle tail wag from elapsed time

Imported fish assets can wait until behavior and water volume logic are proven.

## Water Rendering Follow-Up

Fish can be implemented against the gameplay water query before water rendering is perfect, but the visuals should eventually better match the gameplay volume.

Helpful follow-up tasks:

- Clip/shape lake water meshes to lake footprints instead of rendering full square chunk planes.
- Add debug visualization for water columns, depth, shore distance, and current direction.
- Keep water query and visible water aligned closely enough that fish do not appear outside visible water.

### Phase 2: Fish Prototype

- [x] Add `AnimalSystem` and one fish type.
- [x] Add dynamic runtime fish spawn candidates.
- [x] Add a procedural placeholder fish mesh.

- [x] Despawn fish outside the active radius.

### Phase 3: Player Water State

- [x] Add player water detection.
- [x] Add wading/submerged states.
- [x] Add simple drag and sinking behavior.

### Phase 4: Polish and Interaction

- Improve lake water mesh shape.
- Improve river current direction and speed.
- Add fish flee behavior and small schools.
- Decide whether fishing, catching, or observation-only wildlife belongs in the survival loop.

## Open Questions

- Should fish be purely ambient at first, or eventually a food source?
- Should animals persist if moved/killed, or remain deterministic ambient spawn candidates until interacted with?
- How clear should underwater visibility be in this stylized Idaho summer setting?
- Should the player be able to swim early, or should deep water initially act as a soft boundary/risk?
- How much predator/prey behavior is appropriate for the slow survival fantasy?
